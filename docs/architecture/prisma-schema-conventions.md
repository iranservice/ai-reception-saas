# Prisma Schema Conventions

## Purpose

Define project-level Prisma schema conventions before adding domain models.

## Model Naming

- Prisma models use PascalCase (e.g., `User`, `Business`, `BusinessMembership`)
- Model names should be singular nouns representing a single entity
- Multi-word models use PascalCase without separators (e.g., `BusinessMembership`, not `Business_Membership`)

## Table Naming

- Database tables use snake_case plural names via `@@map`
- Convention: snake_case plural (e.g., `users`, `businesses`, `business_memberships`, `audit_events`)
- Every model must include `@@map("table_name")` to control the database table name

## Field Naming

- Prisma field names use camelCase (e.g., `createdAt`, `userId`, `avatarUrl`)
- Database columns use snake_case via `@map` where the camelCase name differs from snake_case
- Fields that naturally match snake_case (e.g., `email`, `name`, `slug`) do not need `@map`
- Foreign key fields use `@map` to produce snake_case columns (e.g., `userId` ‚Üí `@map("user_id")`)

## ID Strategy

- Use `String @id @default(uuid()) @db.Uuid` for all primary keys
- Do not use autoincrement IDs
- Do not expose internal IDs to public unauthenticated clients
- IDs are generated by the database via `uuid()`

## Timestamp Strategy

- `createdAt DateTime @default(now()) @map("created_at")` on all models
- `updatedAt DateTime @updatedAt @map("updated_at")` on all mutable models
- AuditEvent has `createdAt` only (no `updatedAt` ‚Äî append-only)
- All timestamps are stored in UTC

## Enum Strategy

- Use enums for stable lifecycle states (e.g., `UserStatus`, `BusinessStatus`, `MembershipStatus`)
- Enum names use PascalCase (e.g., `UserStatus`)
- Enum values use SCREAMING_SNAKE_CASE (e.g., `ACTIVE`, `SUSPENDED`, `AI_RECEPTIONIST`)
- Do not use enums for user-configurable concepts (those should be table-backed)
- Roles use enum in MVP (`MembershipRole`); custom roles later require tables

## Relation Strategy

- Use explicit relation names when multiple relations point to the same model
  - Example: `User` has both `createdBusinesses` and `memberships` ‚Äî use `@relation("BusinessCreator")` for clarity
- Avoid ambiguous relations ‚Äî Prisma requires disambiguation when a model references another model multiple times
- Cascade deletes should be avoided by default
- Prefer lifecycle status over destructive deletes
- All relations must specify `fields` and `references` explicitly

## Index Strategy

- Every tenant-scoped model must index `businessId` where applicable
- Add composite indexes for common access paths (e.g., `[businessId, createdAt]` for time-ordered queries)
- Unique constraints must match business rules (e.g., `@@unique([userId, businessId])` on membership)
- Index naming is automatic in Prisma ‚Äî no manual naming needed unless using raw SQL
- Avoid over-indexing ‚Äî only add indexes for known query patterns

## Tenant-Scoped Model Rules

- Tenant-owned records include a `businessId` field
- `User` is global and NOT tenant-scoped
- `Session` is global and NOT tenant-scoped (belongs to User)
- `Business` is the tenant root entity
- `BusinessMembership` links global User to Business ‚Äî it is the gate for tenant access
- Cross-tenant reads are security bugs
- Every query for tenant data must include `WHERE businessId = ?`

## Soft Delete and Status Strategy

- Do not use hard deletes for `User`, `Business`, or `BusinessMembership`
- Use status fields to represent lifecycle transitions:
  - `UserStatus`: `ACTIVE` ‚Üí `SUSPENDED` ‚Üí `DEACTIVATED`
  - `BusinessStatus`: `ACTIVE` ‚Üí `SUSPENDED` / `ARCHIVED`
  - `MembershipStatus`: `INVITED` ‚Üí `ACTIVE` ‚Üí `REMOVED` / `LEFT`
- `Session` uses `revokedAt` timestamp instead of deletion
- `AuditEvent` is append-only ‚Äî no updates, no deletes
- If a generic soft-delete pattern is needed later, use a `deletedAt` timestamp field ‚Äî but prefer status-based lifecycle for domain entities

## JSON Field Rules

- Use `Json?` (nullable) for optional structured metadata
- Do not use JSON for data that needs to be queried, filtered, or indexed
- JSON fields are appropriate for:
  - Audit event metadata
  - Provider-specific response data
  - Flexible configuration blobs
- Document the expected JSON shape in the schema design doc, even though Prisma does not enforce it

## Audit Field Rules

- `AuditEvent` is append-only ‚Äî no `updatedAt`
- Every audit event must include: `actorType`, `action`, `result`, `createdAt`
- Tenant-scoped audit events must include `businessId`
- `actorUserId` is nullable (null for `SYSTEM` and `AI_RECEPTIONIST` actors)
- `metadata` is optional JSON for additional context

## Migration Rules

- One schema task per PR ‚Äî do not combine schema changes with unrelated work
- Migrations must be reviewed before merging
- No broad schema changes without a design doc
- Do not generate migrations in design-only tasks
- Migration names should describe the change (e.g., `add_users_and_businesses`)
- Test migrations locally before pushing

## Forbidden Patterns

- No provider-specific fields in core tenant/identity models (e.g., no `clerkId`, `supabaseUid`)
- No storing raw session tokens ‚Äî only store hashes
- No hard deletes for User, Business, or Membership
- No client-trusted `businessId` ‚Äî always verify via membership
- No direct cross-domain writes ‚Äî use service boundaries
- No Prisma models without tests in implementation tasks
- No `autoincrement()` IDs
- No `@default(cuid())` ‚Äî use `@default(uuid())` with `@db.Uuid`

## Review Checklist

Before merging any Prisma schema change, verify:

- [ ] Model ownership is clear (which domain owns it)
- [ ] Tenant scope is explicit (`businessId` present where needed)
- [ ] Indexes are justified by known query patterns
- [ ] Lifecycle states are documented
- [ ] Audit implications are documented (which actions are audit-relevant)
- [ ] Migration is scoped (single concern per migration)
- [ ] Tests are planned for the new models
- [ ] No forbidden patterns are introduced
- [ ] `@@map` and `@map` are used consistently
- [ ] Relations are explicit and unambiguous
